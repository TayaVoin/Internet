# Вопросы для самопроверки

## 1. Зачем нужны `variables` в GraphQL запросе? Почему нельзя просто склеивать строки?
**Переменные нужны для:**
- Безопасности: склеивание строк приводит к GraphQL инъекциям (аналог SQL инъекций)
- Производительности: сервер может кэшировать разобранный (parsed) запрос, меняются только переменные
- Типобезопасности: переменные строго типизированы в схеме
## 2. Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?
```Стандартный запрос содержит:
{
  "query": "query($id: ID!) { log(id: $id) { message } }",
  "variables": { "id": 123 },
  "operationName": "GetLogById"  // опционально, если несколько операций
}
```
## 3. Почему у GraphQL обычно только один URL endpoint?
- Единый контракт: вся схема доступна через один endpoint
- Гибкость запросов: клиент сам определяет структуру данных
- Упрощение клиентов: не нужно знать множество URL
- Версионирование через схему: изменения управляются на уровне типов, а не URL
## 4. Что означает наличие поля `errors` в ответе сервера? Может ли оно прийти вместе с `data`?
- `errors` означает, что при выполнении запроса произошли ошибки.
- Он может одновременно прийти вместе с `data`, это называется "частичный успех" (partial success).
## 5. Как клиент должен обрабатывать ошибки, если HTTP статус ответа 200?
- Проверить наличие поля errors в JSON ответе
- Проанализировать структуру ошибок (message, path, locations)
- Решить, что делать:
    - Если есть `data` — можно показать частичные данные
    - Если только `errors` — показать сообщение об ошибке
    - Логировать ошибки для отладки
## 6. В чем плюсы использования типизированных клиентов (например, Apollo Client или Relay) перед простыми HTTP-запросами?
- Автогенерация типов на основе схемы (TypeScript/Flow)
- Кэширование результатов запросов
- Нормализация данных (обновление всех ссылок на объект при изменении)
- Оптимистичные обновления (UI обновляется до ответа сервера)
- Пагинация из коробки
- Подписки (Subscriptions) через WebSocket
- DevTools для отладки
