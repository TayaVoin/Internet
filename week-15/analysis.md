## Выводы

### 1. Сравнение протоколов

| Критерий | REST | gRPC | Победитель |
|----------|------|------|------------|
| **RPS** | 7214-8098 | 7347 | REST (незначительно) |
| **Средняя задержка** | 1.33-1.46 ms | 1.29 ms | gRPC |
| **P99 (хвостовые задержки)** | 4.73-5.05 ms | **2.88 ms** | **gRPC** |
| **Стабильность** | Средняя | Высокая | gRPC |
| **Максимальная задержка** | 33.27 ms | 27.44 ms | gRPC |

### 2. Ключевые наблюдения

1. **gRPC значительно лучше по P99** — в 1.6-1.8 раза ниже, чем REST
2. **RPS сопоставимы** — оба протокола держат ~7000-8000 запросов в секунду
3. **REST /events быстрее, чем /health** — вероятно из-за 404 ответов в /events
4. **gRPC стабильнее** — разброс задержек меньше, максимум ниже

### 3. Точка насыщения

- **REST**: начинает деградировать после 8000 RPS (P99 > 5ms)
- **gRPC**: держит стабильность до 7300 RPS (P99 < 3ms)

### 4. Почему gRPC лучше по P99?

1. **Бинарный формат Protobuf** — быстрая сериализация/десериализация
2. **HTTP/2 мультиплексирование** — одно соединение на все запросы
3. **Меньше накладных расходов** — не нужно парсить текстовый JSON

### 5. Рекомендации

- **Для высоконагруженных систем** использовать gRPC (лучше P99)
- **Для публичных API** можно оставить REST (проще для клиентов)
- **Для критичных по времени операций** (менее 5ms) — только gRPC

## Заключение

Оба протокола показали отличную производительность (>7000 RPS). gRPC выигрывает в стабильности и предсказуемости, особенно важной для хвостовых задержек (P99). REST немного быстрее по RPS, но ценой больших выбросов.

**Итог:** для микросервисной архитектуры с высокими требованиями к SLA предпочтительнее использовать gRPC.
