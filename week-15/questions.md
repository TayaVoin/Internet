# Вопросы для самопроверки

## 1. Почему Average Latency (средняя задержка) — плохая метрика? Почему нужно смотреть на P95/P99?
**Проблема Average Latency:**
- Среднее скрывает выбросы (outliers)
- 99% запросов могут быть быстрыми, а 1% — очень медленными, но среднее будет "нормальным"
**Почему важны перцентили:**
- **P50 (медиана)**: типичный пользователь видит такую задержку
- **P95**: 5% пользователей страдают
- **P99**: 1% пользователей в ярости
## 2. Что такое Throughput и в чем он измеряется?
**Throughput (пропускная способность)** — количество запросов, которое система может обработать за единицу времени.
**Единицы измерения:**
- **RPS** (Requests Per Second) — для HTTP/gRPC
- **TPS** (Transactions Per Second) — для БД
- **MB/s** — для передачи данных
## 3. Что такое "точка насыщения" сервера? Как её определить по графику Latency/RPS?
**Точка насыщения (Saturation Point)** — момент, когда система перестает справляться с нагрузкой.
**Признаки насыщения:**
1. RPS перестает расти (достиг максимума)
2. Latency резко увеличивается
3. CPU/Memory упираются в 100%
**Как определить:** До точки насыщения Latency растёт медленно (линейно или чуть быстрее). После — уходит в вертикаль.
## 4. Как размер payload (тела запроса/ответа) влияет на производительность?
| Размер payload | Влияние |
|----------------|---------|
| **Маленький** | Больше RPS, меньше Latency |
| **Большой** | Меньше RPS, больше Latency, выше нагрузка на сеть |

**Почему:**
- Сетевая карта имеет лимит (bandwidth)
- Больше данных = больше времени на передачу
- Больше данных = больше CPU на сериализацию
## 5. Почему нагрузочное тестирование с локальной машины (localhost) не всегда достоверно?
| Проблема | Пояснение |
|----------|-----------|
| **Нет сетевых задержек** | В реальности клиенты далеко, есть RTT |
| **Нет потери пакетов** | TCP ведет себя иначе в плохой сети |
| **Ресурсы общие** | Клиент и сервер борются за CPU |
| **Идеальные условия** | В продакшене всегда хуже |

**Что делать:**
- Тестировать с другой машины (или в облаке)
- Добавлять искусственные задержки (tc, netem)
- Ограничивать ресурсы контейнера (--cpus, --memory)
## 6. Что такое "Coordinated Omission" (координированное упущение) в нагрузочном тестировании?
**Проблема:** Инструменты нагрузочного тестирования часто "пропускают" время, когда сервер тормозит.
**Как это работает (неправильно):**
1. Клиент шлет запрос
2. Ждет ответ
3. Как получил ответ — сразу шлет следующий
**Почему это плохо:**
- Если сервер тормозит (Latency 1с), клиент шлет запросы реже
- RPS автоматически падает
- Тест показывает "нормальную" Latency, хотя сервер умирает
**Как правильно:**
- Шарить запросы с фиксированной частотой (независимо от ответов)
- Использовать инструменты с "open loop" (wrk, ghz это умеют)
