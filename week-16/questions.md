# Вопросы для самопроверки

## 1. Что такое OWASP Top 10? Назовите 3 любые уязвимости оттуда.
**OWASP Top 10** — это список 10 самых критичных рисков безопасности для веб-приложений, обновляемый сообществом OWASP (Open Web Application Security Project).
**Три уязвимости:**
1. **A01:2021 – Broken Access Control** (Нарушение контроля доступа)
   - Пользователь может получить доступ к чужим данным
   - Пример из моего кода: отсутствие проверки владельца заказа
2. **A02:2021 – Cryptographic Failures** (Криптографические ошибки)
   - Передача данных без шифрования, слабое хэширование
   - Пример: хардкод JWT секрета в коде
3. **A05:2021 – Security Misconfiguration** (Ошибки конфигурации)
   - Debug mode включен в продакшене, открытые порты
   - Пример: `--reload` в Dockerfile
## 2. Чем Аутентификация отличается от Авторизации?
| | **Аутентификация (Authentication)** | **Авторизация (Authorization)** |
|---|--------------|-------------|
| **Что это** | Проверка личности | Проверка прав доступа |
| **Вопрос** | "Кто ты?" | "Что тебе можно?" |
| **Пример** | Вход по логину/паролю, JWT токен | Права администратора, доступ к конкретному ресурсу |
| **Когда** | В начале запроса | После аутентификации |
| **Ошибка** | 401 Unauthorized (на самом деле Unauthenticated) | 403 Forbidden |
## 3. Что такое SQL Injection и как ORM помогают от неё защититься?
**SQL Injection** — это атака, при которой злоумышленник внедряет вредоносный SQL-код в запрос через пользовательский ввод.
**Как ORM защищает:**
- Экранирование параметров — ORM автоматически экранирует спецсимволы
- Параметризованные запросы — значения передаются отдельно от кода SQL
- Абстракция — вы работаете с объектами, а не с сырым SQL
## 4. Почему нельзя хранить секреты (API ключи, пароли БД) в коде репозитория?

| **Проблема** | **Последствия** |
| **Git история** | Секрет останется в истории даже после удаления |
| **Доступ** | Все, у кого есть доступ к репозиторию, видят секреты |
| **Компрометация** | При утечке кода утекают и пароли |
| **Смена ключей** | Нужно менять пароли при каждой утечке |
**Где хранить правильно:**
- .env файлы (не в git, добавить в .gitignore)
- Переменные окружения в CI/CD
- Секреты в Kubernetes (Secrets)
- Внешние хранилища (Vault, AWS Secrets Manager)
## 5. Что такое "Man-in-the-Middle" (MitM) атака и как HTTPS защищает от неё?
**MitM (атака "человек посередине")** — злоумышленник перехватывает трафик между клиентом и сервером, подслушивая или подменяя данные.
**Как HTTPS защищает:**
- Шифрование — данные передаются в зашифрованном виде
- Аутентификация сервера — SSL/TLS сертификат подтверждает, что сервер настоящий
- Целостность — нельзя подменить данные без обнаружения
## 6. Зачем нужен принцип "минимальных привилегий" (Least Privilege)?
**Принцип минимальных привилегий** — каждый компонент системы должен иметь ровно столько прав, сколько нужно для работы, и ни граммом больше.
**Зачем это нужно:**
- Ограничение ущерба — при взломе злоумышленник не получит root
- Безопасность по умолчанию — даже если сервис упадет, система выживет
- Соответствие стандартам — PCI DSS, HIPAA требуют этого
