# Вопросы для самопроверки

## 1. Зачем нужен Signaling сервер, если WebRTC — это P2P технология?
Signaling сервер нужен для первоначального "знакомства" пиров. Хотя данные передаются напрямую P2P, пиры должны сначала обменяться мета-информацией:
- SDP (кодеки, возможности)
- ICE кандидаты (адреса, по которым их можно достичь)
Этот обмен нельзя сделать напрямую, потому что пиры еще не знают, как связаться друг с другом. Signaling сервер выступает в роли "почтальона", который помогает им найти друг друга.
## 2. Что такое NAT и почему он мешает прямому соединению?
- **NAT (Network Address Translation)** — механизм, который позволяет нескольким устройствам в локальной сети использовать один внешний IP-адрес.
- **Проблема для P2P**: Устройство за NAT знает свой локальный IP, но не знает своего внешнего IP и порта, через который роутер пропускает трафик. Другой пир не может подключиться напрямую, потому что не знает, какой IP:порт использовать.
## 3. В чем разница между STUN и TURN серверами?
**STUN (Session Traversal Utilities for NAT)**:
- Помогает устройству узнать свой внешний IP и порт
- Работает по принципу "скажи мне, какой у меня IP с твоей стороны"
- Трафик после установки соединения идет напрямую P2P
- Бесплатно, легковесно
**TURN (Traversal Using Relays around NAT)**:
- Релейный сервер, через который идет весь трафик
- Используется, когда NAT слишком строгий и P2P невозможно
- Трафик идет через TURN-сервер (как прокси)
- Дорого (нагрузка на сервер), но гарантирует соединение
## 4. Что такое ICE Candidate?
**ICE Candidate** — это возможный адрес (IP + порт), по которому пир может быть доступен. Устройство собирает несколько кандидатов:
- Локальный (192.168.x.x)
- Внешний (полученный через STUN)
- Релейный (от TURN сервера)
Кандидаты обмениваются через Signaling, и пиры пытаются соединиться, перебирая их пока не найдут работающую пару.
## 5. Почему WebRTC обычно использует UDP, а не TCP?
**UDP**:
- Низкая задержка (нет подтверждений, retransmission)
- Подходит для реального времени (видео, аудио, игры)
- Можно потерять несколько пакетов и ничего страшного
**TCP**:
- Гарантирует доставку, но с задержками
- Retransmission может вызвать "затыки" в видеочате
- Head-of-line blocking
## 6. Что такое SDP (Session Description Protocol) и какая информация в нем содержится?
**SDP** — текстовый протокол для описания мультимедийных сессий. В WebRTC используется для обмена возможностями пиров.
- **Содержит**:
    - Какие медиа-потоки предлагаются (аудио, видео)
    - Какие кодеки поддерживаются (VP8, H.264, Opus)
    - Параметры кодирования (битрейт, частота кадров)
    - Сетевые порты для RTP/RTCP
    - Название сессии, тайминги
