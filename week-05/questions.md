# Вопросы для самопроверки

## 1. В чем главное концептуальное отличие GraphQL от REST?
- REST — это архитектурный стиль, где ресурсы имеют фиксированные эндпоинты (GET /users, POST /users) и сервер определяет, какие данные возвращать.
- GraphQL — это язык запросов, где есть единый эндпоинт (/graphql) и клиент сам решает, какие поля ему нужны.
- **Главное отличие**: контроль над данными переходит от сервера к клиенту.
## 2. Что такое Schema Definition Language (SDL)?
- SDL — это синтаксис для описания GraphQL схемы. На нем определяются типы данных, запросы (Query) и мутации (Mutation).
## 3. Чем Query отличается от Mutation? Могут ли они выполняться параллельно?
- Query — для чтения данных (аналог GET). Могут выполняться параллельно, т.к. не меняют состояние.
- Mutation — для изменения данных (аналог POST/PUT/DELETE). Выполняются последовательно, чтобы избежать конфликтов.
## 4. Что такое "резолвер" (resolver) и кто его вызывает?
- Резолвер — это функция, которая знает, как получить данные для конкретного поля. 
- Ее вызывает GraphQL движок (в моем случае Strawberry) при обработке запроса.
## 5. Как решается проблема N+1 запросов в GraphQL (dataloader)?
- Проблема N+1 возникает, когда для списка из N объектов нужно сделать N дополнительных запросов к БД.
- **Решение**: DataLoader — механизм батчинга и кэширования, который группирует запросы и выполняет их пачками.
## 6. Почему в GraphQL сложно реализовать кэширование на уровне HTTP (как в REST)?
- В REST кэширование работает на уровне URL: GET /users/1 всегда возвращает одного пользователя.
- В GraphQL:
    - Все запросы идут на один URL (/graphql)
    - Тело запроса (query) определяет, какие данные нужны
    - Одинаковые URL могут содержать разные запросы
Поэтому кэширование приходится реализовывать на уровне резолверов или использовать специальные инструменты.
