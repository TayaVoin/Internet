# Вопросы для самопроверки

## 1. Что такое Server Streaming? Приведите пример использования.
**Server Streaming** - это тип gRPC вызова, при котором клиент отправляет один запрос, а сервер возвращает поток сообщений. Соединение остается открытым, и сервер может отправлять данные по мере готовности.
Примеры использования:
- Лента новостей - сервер шлет новые посты
- Биржевые котировки - сервер отправляет обновления цен
- Логи сервера - клиент получает логи в реальном времени
- Прогресс загрузки файла
## 2. Чем Server Streaming отличается от просто получения большого списка (List) в одном ответе?
| Server Streaming | Большой список (List) |
|------------------|----------------------|
| Данные приходят по мере готовности | Нужно ждать весь список |
| Клиент может начать обработку сразу | Клиент ждет последний байт |
| Неограниченный размер | Ограничен размером сообщения |
| Экономия памяти на клиенте | Весь список в памяти |
| Подходит для real-time | Подходит для статичных данных |
## 3. Какие метрики производительности существуют (Latency, Throughput, CPU Usage)?
**Latency (Задержка)** - время от отправки запроса до получения ответа
- Среднее, медиана, P95, P99 (важно для SLA)
- Измеряется в ms
**Throughput (Пропускная способность)** - количество запросов в секунду (RPS)
- Сколько операций система может обработать
- Зависит от параллелизма
**CPU Usage** - загрузка процессора
- Сколько ресурсов потребляет обработка
- Важно для cost optimization
**Другие метрики:**
- Memory Usage (использование памяти)
- Network I/O (трафик)
- Error Rate (процент ошибок)
- Connection count (количество соединений)
## 4. Почему важно переиспользовать канал (Channel) в gRPC клиенте при замерах производительности?
- Установка соединения - создание нового канала требует TCP handshake, что занимает время
- HTTP/2 мультиплексирование - одно соединение может обслуживать множество запросов
- Warm-up - каналы "прогреваются", протокол оптимизируется
Если создавать новый канал на каждый запрос:
- Замеры будут показывать не производительность RPC, а скорость установки соединения
- Результаты будут хуже в 10-100 раз
- Это нечестное сравнение с REST, где тоже используют pool соединений
## 5. В каких сценариях REST может оказаться быстрее или удобнее gRPC?
- REST быстрее когда:
    - Очень маленькие сообщения (накладные расходы Protobuf больше выигрыша)
    - Простые CRUD операции с редкими вызовами
    - Кэширование на HTTP-уровне (CDN, прокси)
- REST удобнее когда:
    - Публичное API для внешних разработчиков (легче отлаживать)
    - Браузерные приложения (нет нативной поддержки gRPC)
    - Прототипирование и отладка (curl, Postman)
    - Схема данных часто меняется (не нужно перекомпилировать клиента)
## 6. Что такое "Head-of-Line Blocking" и как HTTP/2 (и gRPC) решает эту проблему?
- **Head-of-Line Blocking** - проблема HTTP/1.1, когда одно медленное сообщение блокирует все последующие за ним, даже если они готовы раньше.
- Пример:
    - Отправили запросы A, B, C
    - A обрабатывается 1 секунду
    - B готов за 10ms, но ждет пока A закончится
    - C тоже ждет
- Как HTTP/2 решает:
    - Мультиплексирование - несколько потоков в одном соединении
    - Фреймы - сообщения разбиваются на фреймы, которые могут перемежаться

Все запросы передаются параллельно, никто никого не блокирует.
gRPC наследует это от HTTP/2, поэтому может обрабатывать тысячи запросов одновременно на одном соединении без блокировок.
