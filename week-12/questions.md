# Вопросы для самопроверки

## 1. Почему мы создаем Deployment, а не сразу Pod?
**Pod** — это минимальная единица в Kubernetes, но он не перезапускается при падении. Если Pod умрет, он не воскреснет.
**Deployment** — это контроллер, который управляет Pod'ами:
- **Самовосстановление**: Если Pod упал, Deployment создаст новый
- **Масштабирование**: Можно легко изменить количество реплик
- **Обновления**: Поддерживает Rolling Updates (без даунтайма)
- **Откаты**: Можно вернуться к предыдущей версии
| Pod | Deployment |
|-----|------------|
| Не перезапускается | Автоматически перезапускает |
| Нельзя масштабировать | Легко масштабируется (replicas) |
| Нет обновлений | Rolling updates, rollback |
| Подходит для тестов | Для продакшена |
## 2. В чем разница между Liveness и Readiness пробами? Что произойдет, если провалится каждая из них?
| Проба | Назначение | Что проверяет | Действие при провале |
|-------|------------|---------------|----------------------|
| **Liveness Probe** | Жив ли контейнер? | Не зависло ли приложение | Kubernetes перезапустит контейнер |
| **Readiness Probe** | Готов ли принимать трафик? | Загрузились ли все зависимости, подключилась ли БД | Kubernetes уберет Pod из Service (трафик не пойдет) |
## 3. Что такое "requests" и "limits" для CPU и памяти? Почему важно их указывать?
| | **requests** | **limits** |
|---|--------------|------------|
| **Что это** | Минимум ресурсов, гарантированных контейнеру | Максимум ресурсов, которые может использовать контейнер |
| **CPU** | `250m` (0.25 ядра) | `500m` (0.5 ядра) |
| **Memory** | `128Mi` | `256Mi` |
| **Зачем** | Планировщик выбирает ноду с такими ресурсами | Защита от "прожорливых" соседей |
**Почему важно:**
- Безопасность: Один контейнер не съест всю память ноды
- Стабильность: Ядро Linux (OOM Killer) не будет убивать процессы
- Планирование: K8s знает, куда поместить под
- Cost control: В облаке вы платите за использованные ресурсы
## 4. Как Service находит свои Pod-ы? (Подсказка: Labels & Selectors).
Service использует селекторы (selectors) по меткам (labels).
- **Как это работает:**
    - Service смотрит на все Pod'ы в кластере
    - Отбирает те, у которых есть все указанные метки
    - Добавляет их в эндпоинты (endpoints)
    - Трафик равномерно распределяется между ними
- **Проверка:**
```bash
kubectl get endpoints bookings-service
```
## 5. Какой тип сервиса (ClusterIP, NodePort, LoadBalancer) используется для внутреннего общения, а какой — для внешнего доступа?
| Тип | Доступ | Где используется |
|-----|--------|------------------|
| **ClusterIP** (по умолчанию) | Только внутри кластера | Для внутренних сервисов (базы данных, бэкенд для фронтенда) |
| **NodePort** | Снаружи через порт ноды (30000-32767) | Для разработки, отладки |
| **LoadBalancer** | Снаружи через облачный балансировщик | Для продакшена (если в облаке) |
| **Ingress** | Снаружи через HTTP/HTTPS правила | Для продакшена (роутинг по доменам) |
## 6. Что такое "Rolling Update" в контексте Деплоймента?
**Rolling Update** — стратегия обновления приложения без даунтайма.
- Постепенно заменяет старые Pod'ы новыми
- Следит, чтобы количество работающих Pod'ов не падало ниже минимума
- Если что-то пошло не так — останавливает обновление
