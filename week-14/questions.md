# Вопросы для самопроверки

## 1. В чем разница между Continuous Integration, Continuous Delivery и Continuous Deployment?
| Понятие | Что значит | Что происходит |
|---------|------------|----------------|
| **CI (Continuous Integration)** | Разработчики часто сливают код в main | Автоматическая сборка и тесты при каждом коммите |
| **CD (Continuous Delivery)** | Код всегда готов к выкатке в прод | После CI можно выкатить вручную одной кнопкой |
| **CD (Continuous Deployment)** | Каждый прошедший тесты коммит автоматически едет в прод | Полная автоматика, без ручного подтверждения |
## 2. Почему тесты должны запускаться *до* сборки Docker образа (или хотя бы до его пуша)?
**Причины:**
- **Экономия времени**: Если тесты упали, не тратим время на сборку образа
- **Экономия ресурсов**: Не жжем CPU на сборку бесполезного образа
- **Безопасность**: Не пушим в registry битые образы
- **Чистота истории**: В registry только проверенные версии
## 3. Зачем нужно кэширование зависимостей в CI пайплайне?
**Без кэша:** Каждый запуск скачивает все зависимости заново (5-10 минут)
**С кэшем:** Использует уже скачанное из прошлых запусков (30 секунд)
## 4. Что такое "Secret" в контексте CI/CD? Как безопасно передать пароль от Docker Hub в пайплайн?
**Secret** — это переменные окружения, которые хранятся НЕ в репозитории, а в настройках CI системы.
**Как нельзя:**
```yaml
- name: Login to Docker Hub
  run: docker login -u tayavoin -p суперсекретный_пароль  # ❌ НИКОГДА!
```
**Как правильно (в GitHub Actions):**
1. Идем в Settings → Secrets → Actions
2. Добавляем DOCKER_USERNAME и DOCKER_PASSWORD
3. Используем в пайплайне:
```yaml
- name: Log in to Docker Hub
  uses: docker/login-action@v2
  with:
    username: ${{ secrets.DOCKER_USERNAME }}
    password: ${{ secrets.DOCKER_PASSWORD }}
```
**Почему безопасно:**
- Секреты не видны в логах
- Не попадают в репозиторий
- Можно ограничить доступ
## 5. Что такое "Matrix Build" (матричная сборка)?
**Matrix Build** — запуск одного и того же пайплайна с разными параметрами.
**Зачем:**
- Проверить совместимость с разными версиями
- Протестировать на разных ОС (ubuntu, windows, macos)
- Параллельно собирать для разных архитектур (amd64, arm64)
## 6. Почему нельзя хранить конфиги продуктовой среды в репозитории кода (или можно, но осторожно)?
**Проблемы с хранением прод-конфигов в коде:**
- **Безопасность:** Пароли, ключи API — в открытом доступе.
- **Случайное изменение:** Разработчик может закоммитить неправильный конфиг.
- **Разные окружения:** dev/stage/prod конфиги перепутаются.
- **История	Пароль:** останется в git истории даже после удаления.
**Как правильно:**
- Через секреты CI (для паролей)
- Через Helm values (для конфигов)
- Через внешние системы (Vault, AWS Secrets Manager)
- Через переменные окружения
