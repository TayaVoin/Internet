# Вопросы для самопроверки

## 1. Какие критерии вы использовали при разделении монолита на микросервисы (или при проектировании сервисов)?
**Основные критерии:**
1. **Бизнес-логика**: Выделил сервисы по доменам (события, уведомления)
2. **Частота изменений**: Notification Service меняется реже, чем Events Service
3. **Нагрузка**: Events Service под высокой нагрузкой, можно масштабировать отдельно
4. **Команды**: Теоретически разные команды могут вести разные сервисы
5. **Технологии**: Notification Service может быть переписан на Go, если потребуется
## 2. Почему вы выбрали именно этот стек технологий (язык, БД, протоколы)?
| Компонент | Выбор | Почему |
|-----------|-------|--------|
| **FastAPI** | Python | Знаком по курсу, асинхронный, быстрый |
| **PostgreSQL** | Реляционная БД | Транзакции, ACID, надежность |
| **gRPC** | Межсервисный | Скорость, бинарный протокол, строгая типизация |
| **REST** | Внешнее API | Удобно для клиентов, документация (Swagger) |
| **Nginx** | Gateway | Легковесный, быстрый, проверенный |
| **Docker** | Контейнеризация | Везде работает одинаково |
| **GitHub Actions** | CI/CD | Встроен в GitHub, бесплатно |

**Альтернативы:**
- Можно было взять **GraphQL**, но REST проще для финального проекта
- **RabbitMQ** вместо прямых gRPC вызовов, но оверхед
## 3. Как ваши сервисы справляются с ошибками (retry, circuit breaker, graceful degradation)?
| Сценарий | Обработка |
|----------|-----------|
| **Notification Service упал** | Events Service продолжает работу, уведомления сохраняются в БД и отправляются позже (graceful degradation) |
| **gRPC таймаут** | Retry 3 раза с exponential backoff |
| **БД недоступна** | Возвращаем 503, клиент повторяет позже |
| **Превышение нагрузки** | Rate limiting на gateway (slowapi) |
| **Падение сервиса** | Docker/K8s перезапускает контейнер |

**Graceful shutdown:**
```python
import signal
def handle_sigterm(*args):
    server.shutdown()
signal.signal(signal.SIGTERM, handle_sigterm)
```
## 4. Как организован деплой и обновление вашей системы без простоя (Zero Downtime)?
**Стратегии:**
- Rolling update в Kubernetes
    - Постепенная замена подов
    - Healthchecks проверяют готовность
    - Трафик не идет на неготовые поды
- Docker Compose (локально)
    - Просто перезапуск (не zero downtime)
- Blue-Green deployment
    - Две версии (blue и green)
    - Переключение трафика после проверки
## 5. Что было самым сложным в интеграции всех компонентов вместе?
- Сеть между контейнерами
    - Проблема: Сервисы не видели друг друга по имени
    - Решение: Правильная настройка networks в docker-compose
- Согласованность данных
    - Проблема: Notification Service должен знать о новых событиях
    - Решение: gRPC вызовы + fallback (сохранение в локальную БД)
- Версионирование протоколов
    - Проблема: При изменении proto файлов клиенты ломались
    - Решение: Добавил обратную совместимость (optional поля)
