# Вопросы для самопроверки

## 1. Почему в микросервисах нельзя просто использовать `BEGIN TRANSACTION ... COMMIT` для операций, затрагивающих несколько сервисов?
- У каждого микросервиса своя база данных
- Нет единого менеджера транзакций между разными базами
- Двухфазный коммит (2PC) создает проблемы с производительностью и масштабированием
## 2. В чем разница между Оркестрацией и Хореографией? Какой подход проще для старта, а какой лучше масштабируется?
- Оркестрация: центральный сервис управляет всеми шагами (проще для старта)
- Хореография: сервисы общаются через события (лучше масштабируется)
## 3. Что такое "компенсирующая транзакция"? Приведите пример.
- Действие, которое отменяет результаты предыдущего успешного шага
- Пример: если оплата не прошла, отменить резерв товара
## 4. Может ли компенсирующая транзакция завершиться ошибкой? Что делать в таком случае?
- Да, может. Нужно использовать retry, логирование, Dead Letter Queue
## 5. Что такое "согласованность в конечном счете" (Eventual Consistency)?
- Система гарантирует, что при отсутствии новых изменений все копии данных со временем придут к одному значению
