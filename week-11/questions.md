# Вопросы для самопроверки

## 1. Зачем создавать пользовательскую сеть (bridge network) в Docker Compose, если есть дефолтная?
| Дефолтная сеть | Пользовательская сеть |
|----------------|----------------------|
| Сервисы доступны по имени, но с ограничениями | Полный DNS-резолвинг по имени сервиса |
| Все сервисы в одной сети, нет изоляции | Можно создавать несколько изолированных сетей |
| Нельзя отключить сервисы друг от друга | Легко контролировать, какие сервисы видят друг друга |
| Автоматически создается | Явно описывается в compose-файле |

**Зачем нужна пользовательская сеть:**
- **Изоляция**: Можно разделить сервисы на разные сети (например, фронтенд в одной, бэкенд в другой)
- **Контроль**: Только нужные сервисы видят друг друга
- **Читаемость**: Явно видно, кто с кем общается
- **Безопасность**: База данных может быть недоступна извне, но доступна для приложения
## 2. В чем разница между `build` и `image` в описании сервиса?
| `build` | `image` |
|---------|---------|
| Собирает образ из Dockerfile | Использует готовый образ из registry |
| Требует наличия Dockerfile и контекста | Просто скачивает или использует локальный образ |
| Подходит для разработки (часто меняется) | Подходит для готовых сервисов (базы, nginx) |
| Можно комбинировать с `image` для тегирования | Не требует сборки |
## 3. Что делает `depends_on`? Гарантирует ли он, что приложение в зависимом контейнере полностью инициализировалось?
- **depends_on определяет порядок запуска контейнеров:**
    - Сначала запускается зависимый контейнер
    - Потом тот, который от него зависит
- **depends_on не гарантирует**, что приложение в контейнере готово принимать запросы. Он ждет только запуска контейнера (статус "running"), а не готовности сервиса внутри.
## 4. Как работают healthcheck-и и как использовать их в `depends_on`?
- Как работает healthcheck:
    - Docker периодически выполняет команду внутри контейнера
    - Если команда возвращает 0 (success) — контейнер здоров
    - Если команда возвращает ошибку — контейнер нездоров
    - После N неудачных попыток контейнер помечается как unhealthy
- Использование в depends_on:
```
depends_on:
  postgres:
    condition: service_healthy  # ждать, пока healthcheck не станет успешным
```
## 5. Как передать переменные окружения в контейнер через Compose (минимум 2 способа)?
- **Способ 1: Прямо в файле**
```yaml
environment:
  - DB_HOST=postgres
  - DB_PORT=5432
  - DB_USER=postgres
  - DB_PASSWORD=secret
```
- **Способ 2: Через файл .env**
Создать файл .env в той же папке:
```text
DB_HOST=postgres
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=secret
```
В docker-compose.yml:
```yaml
environment:
  - DB_HOST=${DB_HOST}
  - DB_PORT=${DB_PORT}
  - DB_USER=${DB_USER}
  - DB_PASSWORD=${DB_PASSWORD}
```
- **Способ 3: Через env_file**
```yaml
env_file:
  - .env
  - .env.production  # можно несколько файлов
```
- **Способ 4: В командной строке**
```bash
DB_PASSWORD=secret docker-compose up
```
## 6. Почему Docker Compose редко используют в продакшене (хотя это возможно)?
| Причина | Пояснение |
|---------|-----------|
| **Масштабирование** | Compose не умеет автоматически масштабировать сервисы на несколько нод |
| **Обновления без даунтайма** | Нет rolling updates, blue-green deployment |
| **Мониторинг** | Нет встроенного мониторинга и самовосстановления |
| **Распределение** | Работает на одной машине, не умеет в кластер |
| **Сложность** | Для сложных систем YAML становится нечитаемым |

**Где используют:**
- **Разработка** — идеально для локального запуска
- **Тестирование** — быстрый подъем инфраструктуры в CI/CD
- **Staging** — небольшие проекты
- **Продакшен** — только для очень простых проектов на одной машине
**Альтернативы для продакшена:**
- **Kubernetes** — стандарт индустрии
- **Docker Swarm** — встроен в Docker, но менее популярен
- **Nomad** — от HashiCorp
