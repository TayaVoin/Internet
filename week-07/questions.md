# Вопросы для самопроверки

## 1. Зачем нужен файл `.proto`? Почему мы не пишем классы сообщений сразу на Python?
- Файл `.proto` нужен как **единый контракт** между сервисами. 
- Причины:
    - **Языковая независимость**: Один `.proto` файл можно использовать для генерации кода на Python, Go, Java, C++ и других языках. Если бы мы писали классы сразу на Python, клиенты на других языках не смогли бы с нами взаимодействовать.
    - **Строгая типизация**: Protobuf обеспечивает строгую схему данных и проверяет типы.
    - **Оптимизация**: Protobuf сериализует данные в бинарный формат эффективнее, чем ручная сериализация JSON.
    - **Обратная совместимость**: Номера полей позволяют эволюционировать схеме без поломки клиентов.
## 2. Что такое Unary RPC? Чем он концептуально похож на REST запрос?
- **Unary RPC** — это простейший тип gRPC вызова: клиент отправляет один запрос, сервер возвращает один ответ. Это как обычный вызов функции.
- Сходство с REST:
    - REST: `GET /users/1` → сервер возвращает одного пользователя
    - Unary RPC: `GetUser(id=1)` → сервер возвращает одного пользователя
Оба подразумевают модель "запрос-ответ".
## 3. Почему gRPC обычно эффективнее REST (JSON) по процессору и трафику?
- Бинарный формат: Protobuf передает данные в бинарном виде, а JSON — в текстовом. Бинарные данные занимают меньше места и быстрее парсятся.
- HTTP/2: gRPC использует HTTP/2, который:
    - Мультиплексирует запросы
    - Сжимает заголовки
    - Позволяет серверу отправлять данные клиенту без запроса
- Строгая схема: Не нужно парсить и валидировать JSON, структура данных известна заранее.
## 4. Что такое "обратная совместимость" в Protobuf и зачем нужны номера полей (тэги, например `= 1`)?
- Обратная совместимость означает, что новая версия схемы может работать со старыми данными и наоборот.
- Номера полей (тэги) нужны для:
    - Идентификации полей в бинарном формате — вместо имени поля передается его номер (экономит место)
    - Эволюции схемы:
        - Можно добавлять новые поля (с новыми номерами) — старые клиенты их проигнорируют
        - Можно удалять поля, но не использовать их номера повторно
        - Можно менять типы полей, но осторожно
## 5. Что такое Stub (стаб) на клиенте и Servicer на сервере?
- **Stub** — это сгенерированный код на стороне клиента, который выглядит как локальный объект. Клиент вызывает методы стаба, а стаб упаковывает аргументы в Protobuf, отправляет по сети и распаковывает ответ.
- **Servicer** — это базовый класс на стороне сервера, который нужно реализовать. Мы наследуемся от него и пишем реальную логику.
## 6. В каких случаях вы бы выбрали REST, а в каких — gRPC?
**Выбрать REST, когда:**
- API публичное, используется браузерами или мобильными приложениями
- Нужна человеко-читаемость и простота отладки
- Схема данных простая и редко меняется
- Нужно кэширование на HTTP-уровне (CDN, прокси)

**Выбрать gRPC, когда:**
- Микросервисная архитектура (общение между внутренними сервисами)
- Высокие требования к производительности и низкой задержке
- Нужна строгая типизация и контракты
- Потоковая передача данных (streaming)
- Мультиязычная среда (сервисы на разных языках)
